# 玄衡 README

> 本文档整理当前仓库内与“玄衡”相关的**现行工作流、硬规则、修改步骤、验证方式、边界限制、历史归档**，用于后续严格执行 `assets/puzzles.json` 的谜题优化与分数异常路由。
>
> 目标：当用户说“玄衡”时，不再按普通“改 hints”理解，而是进入一套**有路由、有闭环、有硬闸、有验证、有统一汇报口径**的严格流程。

---

## 1. 玄衡是什么

在这个仓库里，“玄衡”不是昵称，也不是宽泛的“帮我认真一点”。

“玄衡”是一个**严格触发词**：一旦用户在本仓库里说“玄衡”，就意味着后续任务必须切换到**玄衡统一工作流**，按当前仓库里已经存在的技能、规则、验证脚本和闭环要求执行，不能随意简化，不能跳步，不能凭主观感觉汇报“差不多好了”。

其核心特点是：

1. **先路由，再执行**：先判断任务属于“题库/分类优化”还是“分数异常诊断”，不能混成一个模糊流程。  
2. **闭环优先于速度**：不允许“修了一点就先告诉用户”。  
3. **规则优先于灵感**：先过硬规则，再谈自然度微调、梯度、维度和体验。  
4. **当前答案闭环后，才能进入下一个答案**。  
5. **整类全部闭环后，才能向用户汇报分类完成**。  
6. **没有证据时宁可保守，不得编造**。  

一句话概括：

> 玄衡 = 严格路由 + 逐词闭环 + 批量闭环 + 硬规则优先 + 改后必检 + 零错误才可报完成。

## 3. 任务路由：先判定任务类型

玄衡流程的第一步不是改词，而是**判定任务到底属于哪一类**。

### 3.1 A 类：题库 / 分类优化任务

当用户目标是以下类型时，直接进入**题库优化流**：

- “优化 xx 分类”
- “修 hints”
- “清理泄露”
- “调自然度”
- “做整类闭环”
- “按玄衡规则改 `puzzles.json`”
- “处理某个分类的重复、模板词、维度问题、锚点问题”

此时：

- 直接进入 `assets/puzzles.json` 工作流
- **不要默认先做分数链路诊断**
- **不要一上来先查 calibration / controller / override**
- 默认改动边界是 `assets/puzzles.json`

### 3.2 B 类：分数异常诊断任务

当用户目标是以下类型时，进入**分数异常诊断流**：

- “为什么这个词分数这么高 / 这么低”
- “这个百分比不合理”
- “是不是 calibration 抬分了”
- “为什么无关词会高分”
- “这个语义分看起来反常”
- “是不是后处理规则把它抬高了”

此时必须按顺序检查：

1. [data/manual_similarity_overrides.json](../data/manual_similarity_overrides.json)  
2. [data/semantic_calibration_v27_semreal_anchor.json](../data/semantic_calibration_v27_semreal_anchor.json)  
3. [lib/controllers/guess_game_controller.dart](../lib/controllers/guess_game_controller.dart)  
4. 仅在前 3 项排除后，才考虑 [assets/puzzles.json](../assets/puzzles.json)

### 3.3 C 类：`工作` 分类细修子流程

只有在以下条件全部满足时，才允许进入 `工作` 分类的“细书面味精修”子流程：

- 当前目标分类是 `工作`
- 主闭环已经完成
- 明显模板词、模型味、残片、字面泄露、重复等问题已清掉
- 当前分类整类复验已达零错误基线
- 用户进一步要求“继续细修”“继续深抠”“处理残余书面味”“继续精细化”

这时才使用：

- [.github/skills/work-subtle-naturalization/SKILL.md](../.github/skills/work-subtle-naturalization/SKILL.md)

### 3.4 中途切换到诊断流的条件

即便任务起点是“优化分类”，处理中如果发现以下情况，也要**明确告知并切换诊断视角**：

- 很多题 hints 看起来没问题，但分数普遍异常
- 某些高分更像 calibration 中段抬升
- 后处理规则把某类词系统性抬高 / 压低
- override 明显影响体验

此时不能把所有异常都直接归因到 `puzzles.json`。

## 4. 玄衡硬门槛总表

只要用户喊“玄衡”，以下全部视为**硬门槛**，不是建议项。

### 4.1 结构与数量硬门槛

1. 每题必须有 **7 条 hints**。  
2. 任意 hint 必须是**自然语言单元**，不能是空串、碎片、读不通的残片。  
3. 常规长度以短词短语为主，通常不超过 **5 字**；当前严格脚本的自动硬闸长度范围为 **2-6 字**，见 [scripts/xuanheng_check_answer_strict.py:18-20](../scripts/xuanheng_check_answer_strict.py#L18-L20)、[scripts/xuanheng_check_answer_strict.py:193-200](../scripts/xuanheng_check_answer_strict.py#L193-L200)。  
4. 禁止所有 hints 强制整齐化，例如全都四字、全都同长度。严格脚本会直接检查“全四字风格”和“全等长”问题，见 [scripts/xuanheng_check_answer_strict.py:174-179](../scripts/xuanheng_check_answer_strict.py#L174-L179)。

### 4.2 字面隔离硬门槛

5. 任意 hint 不得与 answer 共享任何字符。  
6. 这是最基础的自动硬闸之一；一旦重字，当前答案立即判失败。

### 4.3 唯一性硬门槛

7. 同题内 7 条 hints 不能重复。  
8. 同分类内不能与其他谜题复用相同 hint。  
9. 全局不能与其他谜题复用相同 hint。  
10. 禁止近义改写却语义骨架相同的“伪去重”。
11. **禁止 7 条 hints 中出现同义词/近义词替换**：例如"郁闷"与"郁结"不能同时出现，命中即算重复，必须改写。

### 4.4 前 6 条泄露硬门槛

11. **前 6 条 + 已知分类不得直接锁定答案**。  
12. 用户如果明确指出第 2/3/4/5/6 条某一槽位已经明显泄露答案方向，这就是高优先级违规信号。  
13. 第 5/6 条如果已经显著缩小答案空间，也应优先降强度后复验。  
14. 任何“分类 + 前 6 条已可高置信秒猜”的情况，必须回退，不能放行。

### 4.5 第 7 条锚点硬门槛

15. 第 7 条必须是最强锚点。  
16. 第 7 条应与答案机制直连，承担最终收束职责。  
17. 若第 6/7 条只是泛词、弱相关词、假锚点，必须替换。  
18. 不允许出现“第 5 条比第 7 条更强”“第 7 条很虚而前面更像答案”的倒挂情况。

### 4.6 强度阶梯硬门槛

19. 7 条 hints 的目标强度应大体符合：

| 槽位 | 目标强度 |
|------|----------|
| slot 1 | 30% |
| slot 2 | 40% |
| slot 3 | 50% |
| slot 4 | 60% |
| slot 5 | 70% |
| slot 6 | 80% |
| slot 7 | 90% |

20. 这里的百分比不是字符串相似度，而是：

> 已知分类时，玩家第一次看到“分类 + 当前 hint”，自然联想到答案的概率强度。

21. 允许自然语言上的轻微非机械抖动，但**不允许明显跳跃、倒挂、提前锁词**。  
22. 若为了机械贴 30/40/50/60/70/80/90 而牺牲自然表达，这也不合格。  
23. `--review-json` 闭环审查要求目标强度正好写成 `[30, 40, 50, 60, 70, 80, 90]`，见 [scripts/xuanheng_check_answer_strict.py:18](../scripts/xuanheng_check_answer_strict.py#L18)、[scripts/xuanheng_check_answer_strict.py:146-151](../scripts/xuanheng_check_answer_strict.py#L146-L151)。

### 4.7 维度分散硬门槛

24. 7 条 hints 必须尽量覆盖 **7 个不同主维度**。  
25. 相邻槽位不得重复同一主维度。  
26. 全部 7 条中，同一主维度只允许出现 1 次。  
27. 近义改写但语义骨架相同，仍然算维度重复。  
28. `--review-json` 闭环审查会检查维度字段是否重复，见 [scripts/xuanheng_check_answer_strict.py:121-129](../scripts/xuanheng_check_answer_strict.py#L121-L129)。

常见维度示例：

- 时间相位
- 空间方位
- 典型场景
- 典型对象 / 人物
- 动作 / 机制
- 功能 / 用途
- 结果 / 反馈
- 情绪 / 氛围
- 观察视角
- 搭配 / 文化联想

### 4.8 自然语义硬门槛

29. 每条 hint 都应自然、完整、可单独理解。  
30. 优先真实母语表达，不能像说明书、模板词池、压缩残句。  
31. 不得为了整齐好看而统一字数、统一句式、统一四字格。  
32. 应优先使用简洁“标签化”短词 / 短语，而不是“人物是… / 发生在… / 作用是…”这类包裹型说明句。  
33. 必须避免明显模型味、元描述腔、说明文腔、流程占位词。

### 4.9 禁止模板词 / 元描述 / 泛词 / 残片

34. 禁止元描述词、模板词、流程占位词，如：
   - “关键线索”
   - “场景线索”
   - “阶段提示”
   - “答案映射”
   - “主要特征”
   - “典型特征”
   - 以及类似“剧情铺垫 / 事件推进 / 过程细节 / 情境线索”等元话语
35. 严禁可泛化到大量答案的空泛词。  
36. 严禁不完整残片，例如依赖额外脑补才能成立的拼接词。  
37. 自动脚本里已有一批模板片段硬检，见 [scripts/xuanheng_check_answer_strict.py:21-37](../scripts/xuanheng_check_answer_strict.py#L21-L37)、[scripts/xuanheng_check_answer_strict.py:210-217](../scripts/xuanheng_check_answer_strict.py#L210-L217)。

### 4.10 禁止主观臆造

38. 没有现实依据、常识依据、网络常见表达依据的词，不能写入。  
39. 互联网资料可作为辅助理解和选词参考，但不是拿来机械搬运，也不是每条都必须附出处。  
40. 若证据不足，宁可中性降强度，也不编造。

### 4.11 禁止未检报检

41. 任何修改后都必须重检。  
42. 未重检就说“检查过了”“已通过”“已闭环”，属于流程违规。  
43. 当前答案没有完成“改后回读 + 当前答案重检 + 结构化复核”，就不能算闭环。  
44. 整个分类只要还有一个答案没闭环，就不能说“该分类已完成”。

## 5. 修改 `assets/puzzles.json` 的具体步骤

这一节按玄衡实际执行顺序写，不按抽象理论写。

### 5.1 第一步：先确定边界，只改 `assets/puzzles.json`

默认情况下，玄衡分类优化的修改边界是：

- 目标文件：[assets/puzzles.json](../assets/puzzles.json)
- 默认不碰模型训练
- 默认不碰 calibration
- 默认不碰 controller
- 默认不碰 Flutter UI

只有当任务已经被判定为“分数异常诊断”，或者处理中发现系统层原因，才扩展到其它文件。

### 5.2 第二步：定位目标分类或目标答案

进入修改前，先明确：

- 目标是单题还是整类
- 是按 `category` 处理，还是只修一个 `answer`
- 是否存在用户明确点名的问题槽位

如果用户说的是“优化某分类”，必须先把该分类下所有目标答案找出来，不能边找边随意汇报。

### 5.3 第三步：回读当前答案原始数据

对当前正在处理的答案，先回读其原始结构：

- `answer`
- `category`
- 当前 7 条 hints

目的不是抄出来，而是理解当前题到底有哪些问题、哪些槽位还能保留、哪些槽位已经明显违规。

### 5.4 第四步：先做第一轮硬规则检查

在改词前，先对当前答案做第一轮规则扫描，至少覆盖：

- 是否与 answer 共字
- 是否同题重复
- 是否存在模板词 / 元描述 / 泛词 / 残片
- 是否同类复用 / 全局复用
- 是否有前 2-6 条过强
- 是否高位锚点无效
- 是否维度重复
- 是否整体自然度不足
- 是否存在强制整齐化痕迹

### 5.5 第五步：问题分型

玄衡不会“看到不好就重写”，而是先给问题分型。

常见问题类型：

| 问题类型 | 说明 |
|----------|------|
| 前位泄露 | 第 2-6 条某条过强，提前暴露答案方向 |
| 高位锚点无效 | 第 6/7 条太虚、太泛、不能收束 |
| 维度重复 | 多条 hints 实际在讲同一维度 |
| 自然度不足 | 像模板词、说明句、拼接残片 |
| 字面污染 | hint 与 answer 共字 |
| 复用问题 | 同题 / 同类 / 全局重复 |
| 强度失衡 | 某槽位明显强于其后槽位，或中途跳跃 |
| 风格失控 | 7 条整体质量差，局部修补收益很低 |

### 5.6 第六步：优先最小改动，不默认整组重写

玄衡的默认策略是：

> 能改单槽，就不重写整组。  
> 能局部修复，就不大面积推倒重来。

优先动作：

1. 若第 2-6 条某槽位过强：优先下调该槽位强度。  
2. 若第 6/7 条是假锚点：优先替换高位锚点。  
3. 若两个槽位维度撞车：优先改单个槽位，让维度分开。  
4. 若只有一个 hint 与 answer 共字：优先只替换该 hint。  
5. 若只是复用问题：优先替换冲突 hint，而不是全部重写。  

### 5.7 第七步：只有在这些情况下才重写整组

只有以下场景才考虑整组重写：

- 7 条 hints 整体风格失控
- 多条维度重复或同义词重复
- 阶梯整体错乱
- 模板词 / 残片 / 重复 / 泄露是系统性的
- 单槽修补已无法收口
- 当前题整体质量太差，局部修补收益很低

### 5.8 第八步：写新 hint 的具体口径

写新 hint 时，必须同时满足：

1. 不与答案共字  
2. 是自然短语，不是残片  
3. 尽量落在独立主维度上  
4. 强度与所在槽位相匹配  
5. 不与同题 / 同类 / 全局重复  
6. 不属于模板词 / 元描述 / 泛词  
7. 有常识或网络常见表达支撑  
8. 不为了工整牺牲自然性

额外要求：

- **不得从提示词池里批量抽词填空**  
- **每个答案的 7 个提示词都要单独构思**  
- **本地模型不能作为 hint 候选生成、排序、打分或写入依据**

### 5.9 第九步：改完后第一优先级不是继续想下一个词，而是回读当前答案

只要当前答案有任意 1 条 hint 被修改，下一步必须先做：

1. 回读当前答案 7 条 hints  
2. 立即重检当前答案硬规则  
3. 先确认刚改的当前词没有制造新的违规  
4. 当前答案未闭环前，不得进入下一个答案

这条规则非常关键：

> 修改后的第一优先级不是“继续想更好的词”，而是“证明当前答案没有违反硬规则”。

### 5.10 第十步：修改后先过四项基础硬闸

根据当前 skill 的永久规则，只要当前答案有改动，修改后的下一步必须至少先重检：

- 字面隔离
- 长度
- 元描述 / 模板词
- 同题重复

这 4 项任意 1 项未过：

- 禁止继续讨论强度
- 禁止继续讨论维度
- 禁止继续讨论自然度微调
- 禁止进入下一个答案

### 5.11 第十一步：完成当前答案的完整闭环

当前答案要算“闭环完成”，至少要满足：

1. 当前答案 7 条已回读  
2. 硬规则已检查  
3. 修改已经完成  
4. 修改后已复检  
5. 没有明显未解决违规项  
6. 已打印 / 记录当前答案 7 条 hints 的逐条信息：
   - hint 文案
   - 所属维度
   - `分类 + hint` 的可猜度百分比判断
7. 若要宣称“玄衡闭环通过”，必须补充 `--review-json` 结构化审查

若当前答案尚未打印 7 条 hints 的维度和可猜度判断，则该答案不算闭环。

### 5.12 第十二步：当前答案闭环后，才允许进入下一个答案

这一条是玄衡执行纪律的核心：

- 不允许“整类粗修一轮，再回头查”
- 不允许“先都改一遍，最后统一补检查”
- 不允许“先把语义调顺，再回头补硬规则”

固定顺序只能是：

1. 只处理当前一个答案  
2. 当前答案检查 → 修改 → 当前答案重检  
3. 当前答案零错误  
4. 记录维度与可猜度  
5. 进入下一个答案

## 6. 分类修复标准工作流

这部分回答“你怎么进行分类修复的就怎么写”。

### 6.1 总原则：分类任务必须整类闭环

只要用户说“优化某分类”，就不是修单题，而是进入**整类闭环模式**：

- 找出该分类下全部目标谜题
- 对每一个答案逐词闭环
- 全部答案都闭环后，再统一验证
- 全部验证通过后，再统一向用户汇报

禁止：

- 只修 1 题就说“分类已完成”
- 中间修一部分就模糊汇报“差不多好了”
- 把“单题修好”当成“整类结束”

### 6.2 标准执行顺序

#### 阶段 1：找出当前分类全部答案

先把该分类的全部答案列出来，形成明确处理范围。

#### 阶段 2：只处理当前一个答案

一次只看一个答案，不并行粗改多题。

#### 阶段 3：对当前答案做全量检查

检查项必须覆盖：

- 字面隔离
- 元描述 / 模板词
- 长度
- 同题重复
- 同类重复
- 全局重复
- 前 6 条不过强
- 第 7 条最强
- 7 维度全异
- 自然语义
- 不可主观臆造

#### 阶段 4：若有违规，只修当前答案

此时只针对当前答案落笔，不跳到其他题，也不先整类平推。

#### 阶段 5：修完后立即重检当前答案

修改后的下一步只能是**当前答案重检**，不是继续修别题。

#### 阶段 6：当前答案零错误后，记录逐条信息

必须记录 / 打印：

- 7 条 hint 文案
- 每条所属维度
- 每条的 `分类 + hint` 可猜度百分比判断

#### 阶段 7：进入下一个答案

只有当前答案闭环后，才能进入下一个答案。

#### 阶段 8：全部答案闭环后，再做整类验证

整类中任何一个答案没闭环，都不能进入“分类完成”汇报阶段。

### 6.3 分类修复时的优先级口径

分类修复不是所有问题一视同仁，优先级一般如下：

1. **硬失败项优先**：共字、同题重复、模板词、长度错误  
2. **前位泄露优先**：第 2-6 条过强  
3. **高位锚点无效优先**：第 6/7 条虚弱或泛化  
4. **维度重复优先**：7 维度全异未达成  
5. **同类 / 全局复用优先**  
6. **自然度精修最后**

### 6.4 分类修复后的统一汇报格式

玄衡风格的汇报不说“改好了”就结束，而应至少包含：

#### 修改范围

- 改了哪个分类
- 涉及多少题
- 是否整类闭环

#### 发现的问题

- 前 6 条泄露
- 锚点不足
- 维度重复
- 模板词
- 字面污染
- 复用问题
- 自然度问题

#### 修改策略

- 哪些题是局部修
- 哪些题是整组重构
- 哪些题只是降强度 / 重排
- 哪些题保留不动

#### 验证情况

- 单题检查是否通过
- 分类检查是否通过
- 全局规则是否通过
- 是否还有残留问题

#### 未完成项

若存在，必须明确：

- 哪些没完成
- 为什么没完成
- 下一步怎么做

禁止使用含糊话术糊弄过去。

## 7. 分数异常诊断工作流

如果用户要的不是“改分类”，而是“查分数异常”，必须走另一条路。

### 7.1 标准排查顺序

按固定顺序检查：

1. [data/manual_similarity_overrides.json](../data/manual_similarity_overrides.json)  
   看是否命中答案-猜词人工覆盖。  
2. [data/semantic_calibration_v27_semreal_anchor.json](../data/semantic_calibration_v27_semreal_anchor.json)  
   看 raw -> calibrated 是否出现中段抬升。  
3. [lib/controllers/guess_game_controller.dart](../lib/controllers/guess_game_controller.dart)  
   看 cap / floor / penalty / 业务规则后处理是否影响最终分。  
4. [assets/puzzles.json](../assets/puzzles.json)  
   仅在前三者排除后，才把问题归到 hints 文案层。

### 7.2 项目经验值

当前仓库已有明确经验提醒：

- raw 60 常映射到约 71
- raw 70 常映射到约 80

也就是说，很多用户体感上的“离谱高分”，其实可能来自 calibration 中段抬升，而不是 hints 本身过强。

### 7.3 诊断类输出格式

分数异常诊断时，应按以下顺序输出：

1. **结论**：异常更像来自哪一层  
   - 覆盖 / 校准 / 后处理 / 题库  
2. **证据**：至少给出一条规则命中或映射证据  
3. **动作建议**：  
   - 改规则  
   - 改校准  
   - 改题库  
   - 暂不改

### 7.4 诊断流禁止事项

- 未排查 calibration 前直接改题库
- 用一次猜测结果下全局结论
- 为了追求表面单调性而牺牲自然表达
- 未执行检查就宣称“已通过”

## 8. 验证命令与通过标准

玄衡流程中，“验证”不是一句口头话，而是明确的脚本与通过条件。

### 8.1 单答案硬闸

命令：

```bash
python3 scripts/xuanheng_check_answer_strict.py <答案>
```

作用：

- 检查当前答案的 7 条 hints 是否存在自动硬闸错误
- 覆盖：数量、全四字 / 全等长、同题重复、长度、字面重叠、模板片段、全局复用、同类复用、early reveal 等

命令帮助见：

- [scripts/xuanheng_check_answer_strict.py:1-20](../scripts/xuanheng_check_answer_strict.py#L1-L20)
- `python3 scripts/xuanheng_check_answer_strict.py --help`

### 8.2 单答案玄衡闭环验证

若要宣称当前答案“玄衡闭环通过”，不能只跑上面的硬闸，还必须补充：

```bash
python3 scripts/xuanheng_check_answer_strict.py <答案> --review-json <review.json> --require-review
```

其中 `review.json` 必须至少包含：

- `answer`
- `rows`（共 7 项）
- 每项包含：
  - `hint`
  - `dimension`
  - `target_strength`
  - `basis`
  - `early_lock_risk`

脚本会检查：

- `answer` 是否匹配
- `rows` 是否恰好 7 条
- 每条 `hint` 是否与当前题一致
- `dimension` 是否缺失 / 重复
- `target_strength` 是否严格等于 `30,40,50,60,70,80,90`
- `basis` 是否缺失
- 前 6 条的 `early_lock_risk` 是否明确

对应代码见：

- [scripts/xuanheng_check_answer_strict.py:78-152](../scripts/xuanheng_check_answer_strict.py#L78-L152)

### 8.3 分类级检查

命令：

```bash
python3 xuanheng_check_detailed.py
```

作用：

- 做兼容性的分类级详细检查
- 当前脚本实际内容是**针对“成语”分类**的详细检查，不是通用全分类脚本，见 [xuanheng_check_detailed.py:33-37](../xuanheng_check_detailed.py#L33-L37)

因此在文档和汇报中，应把它描述为：

- **分类级兼容补充检查脚本**  
- 不是唯一权威，也不是所有分类通用闭环依据

### 8.4 全局规则检查

命令：

```bash
python3 scripts/validate_global_hint_rules_v1.py
```

作用：

- 对全局题库做规则扫描
- 检查：
  - 非 7 条 hints 条目
  - 同题重复
  - 强制全四字
  - 元描述 / generic term
  - 字面重叠
  - early reveal
  - 全局复用
  - 同类复用
- 生成报告：
  - `tmp/global_rules_v1/global_hint_rules_v1_report.json`

相关代码见：

- [scripts/validate_global_hint_rules_v1.py:14-169](../scripts/validate_global_hint_rules_v1.py#L14-L169)

### 8.5 可猜度 / 强度展示（按需）

命令：

```bash
python3 scripts/score_category_semantic_guess_v1.py --category <分类>
```

可选参数：

- `--input`
- `--model-path`
- `--calib-path`
- `--out-dir`
- `--temperature`

作用：

- 展示某个分类内各 hint 的语义强度 / 猜测概率分布
- 用于辅助说明“哪条过强、哪条过弱、哪个答案前 6 条已接近锁词”
- 它是**辅助展示工具**，不是替代硬规则的总裁判

### 8.6 旧脚本的定位

命令：

```bash
python xuanheng_check_answer.py <答案>
```

定位：

- 只能作为兼容补充
- **不能再作为闭环依据**
- 单答案闭环宣称必须以严格脚本 + `--review-json` 为准

### 8.7 通过标准

#### 单答案“自动硬闸通过”

至少表示：

- 没有自动硬闸 issues
- 当前答案基础结构合格

#### 单答案“玄衡闭环通过”

必须同时满足：

- 自动硬闸通过
- `--review-json` 审查通过
- 7 条 hints 已逐条给出维度与目标强度判断
- 前 6 条风险已被明确复核

#### 整类“闭环完成”

必须同时满足：

- 该分类下所有答案都已逐词闭环
- 没有任何答案跳过回读 / 重检 / review 记录
- 分类级检查完成
- 按需完成全局规则检查

#### 全局“规则通过”

必须同时满足：

- `validate_global_hint_rules_v1.py` 无 hard fail
- 没有 early reveal 硬失败
- 没有 global/category reuse 等机器硬失败项

## 9. 分类修复时的写作原则

这一节回答“改 hint 时应该怎么写”。

### 9.1 强度阶梯优先，但不机械贴点

- slot1 到 slot7 应该是从弱到强的自然递进
- 不能为了“看起来对齐”而牺牲自然语义
- 真正的标准是玩家看到“分类 + hint”时的自然联想概率

### 9.2 前 6 条是铺路区，不是定锤区

- 前 6 条的职责是缩小空间
- 不应在第 2-6 条任意某条突然直冲答案
- 若已经“看分类 + 前 6 条就差不多只剩一个答案”，必须回退

### 9.3 第 7 条必须承担最终收束

- 第 7 条必须最强
- 必须是真锚点，而不是一个大而虚的词
- 优先与答案机制直连

### 9.4 7 维度全异

- 不是换近义词就算新维度
- 不是都写场景、都写对象、都写效果
- 每条都应尽可能落在不同的主维度上

### 9.5 自然语义优先于整齐化

- 通常控制在 5 字左右
- 必要时可放宽到 6 字
- 不强制统一四字格
- 不统一句式
- 不统一节奏
- 不为了“整齐”写生硬词

### 9.6 简洁标签化优先

更倾向于：

- 自然短标签
- 可单独理解的短词 / 短语

而不是：

- “发生在……”
- “人物是……”
- “表示一种……”
- “属于某种……”

### 9.7 可以用网络，但不能依赖本地模型写 hints

允许：

- 通过网络搜索词典 / 百科 / 高频用法辅助理解
- 验证某表达是否真实、常见、自然

不允许：

- 用本地模型生成 hint 候选
- 用本地模型对候选排序
- 用本地模型分数当写入依据
- 用本地模型兜底判断“这个词应该可以吧”

### 9.8 不知道就不要乱写

- 证据不足时宁可保守
- 想不出干净替换点时，允许跳过当前槽位或跳题
- 尤其在 `工作` 分类细修阶段，找不到一槽干净收口点时，不应硬凿

## 10. 特殊情况与中止条件

玄衡流程并不是“无论如何都要改到看起来更花哨”，遇到下面情况应中止并说明。

### 10.1 分类定义本身有问题

- 当前题其实不适合这个 category
- category 边界本身含混
- 该题放错分类导致怎么写都怪

### 10.2 问题其实属于系统层

- override 造成的体验异常
- calibration 中段抬升
- controller 后处理抬分 / 压分
- 语义问题并不在 hints 层

### 10.3 缺乏可靠语义依据

- 再改就会开始主观脑补
- 找不到现实语用依据
- 找不到常识支持
- 候选词都不自然 / 不准确 / 不稳

### 10.4 用户目标彼此冲突

典型冲突：

- 要“极自然”
- 又要“极难秒猜”
- 还要“每条都独特且高辨识”

这些目标在某些答案上不能同时最大化，此时应明确说明 trade-off。

### 10.5 `工作` 分类细修的停机条件

在 `工作` 分类的细书面味精修阶段，满足任一条件即可停止：

- 剩余命中项全部属于合理业务术语
- 再改只会降低准确度或破坏梯度
- 已经没有干净的一槽替换点
- 用户停止或切换任务

### 10.6 禁止过度优化

如果某些题已经很好，就不应该为了“显得我在做事”而硬改：

- good case 保留不动
- 局部有问题就局部修
- 不为了统一风格而抹掉已有好词

## 11. `工作` 分类子流程：细书面味精修

这个子流程只针对 `工作` 分类，而且只在主闭环结束后才允许进入。

### 11.1 启用前提

必须同时满足：

- 目标分类是 `工作`
- 主闭环已完成
- 明显模板词、残片、泄露、重复等已清空
- 当前分类整类复验为零错误
- 用户明确要求继续细修 / 深抠 / 精细化

### 11.2 核心原则

1. 这是**细修技能**，不是重写技能。  
2. 优先一题一槽最小改动。  
3. 找不到干净的一槽替换点，就跳题。  
4. 必须保持梯度不被破坏。  
5. 合理业务术语不乱改。  
6. 每次只处理一个答案；每次只改一个槽位或最小相邻槽位集合。

### 11.3 推荐扫描词

细修时可用轻量词面扫描找候选，但“命中”不等于必须改。当前技能文件中给出的示例包括：

- `路径`
- `思路`
- `功能`
- `记录`
- `整理`
- `计划`
- `表`
- `进度`

### 11.4 已验证有效的细修模式

当前技能文件给出的经验示例：

- `时间表` -> `什么时候做`
- `结果上呈` -> `把结果交上去`
- `阶段汇总` -> `一段小结`
- `核对` -> `仔细过下`
- `留个记录` -> `记下来`
- `进度记录` -> `把事记下`
- `要点整理` -> `挑几条说`
- `进度梳理` -> `把进展讲讲`
- `回归` -> `再跑一轮`

这些不是模板库，而是已经验证过的“细书面味 -> 更自然口语”的替换经验。

### 11.5 明确可接受的术语

以下类型默认先判为可接受业务词，除非用户明确点名：

- 项目管理术语：`预算`、`批复`、`可研`、`计划`
- 审核 / 质量术语：`合规`、`核对`、`回归`
- 常见工作动作词：`做个表`、`翻记录`、`验功能`

### 11.6 输出要求

细修阶段的汇报至少要说明：

- 本轮改了哪个答案、哪个槽位
- 为什么这个词属于“细书面味”而非必要术语
- 当前答案验证结果：
  - 当前题规则检查
  - 梯度是否仍保持递进

## 12. 历史归档：2026-05-26 hints 全量优化

这部分来自已读取的项目记忆归档，用于保存玄衡在本仓库中的一次重要实践结果。

### 12.1 背景

在 2026-05-26，对 [assets/puzzles.json](../assets/puzzles.json) 全部 23 个分类、878 个谜题、6,146 条 hints 做过一次系统性优化。

当时识别出的主要问题包括：

- 模板生成污染
- 跨分类乱码 / 污染
- 同题 / 同类重复
- 强制四字化
- 大量缺乏真实可猜度的占位词

### 12.2 当时采用的优化标准

整合玄衡硬规则 + 市面 AI 猜词游戏实践后，归档中的标准是：

1. **字面隔离**：hint 不包含答案任意字符  
2. **7 维度分散**：7 条 hint 对应 7 个不同主维度  
3. **可猜度阶层**：30% → 40% → 50% → 60% → 70% → 80% → 90% 严格递增  
4. **自然短语风格**：纯名词 / 动词 / 形容词，2-5 字自然变化，零解释句式  
5. **零重复**：同题内、同类目内、全局均不重复

### 12.6 设计迭代记录

归档里保留了以下迭代经验：

- v1：全部强制四字 -> 被判违反“禁止强制四字化”  
- v2：自然语言但太口语 -> 用户要求参考更像短标签的风格  
- v3：长句解释型 -> 用户要求回到简洁短语  
- v4：自然度更好，但个别 slot 提前过强  
- vFinal：逐题精准校准可猜度阶层 + 自然短词 + 零重复

这些经验说明：

> 玄衡不是“越口语越好”，也不是“越工整越好”，而是**自然、分散、递进、不过早锁词、最终能收束**的平衡。

## 13. 最终执行摘要

当用户说：

> “玄衡，帮我优化 xxx 分类”

就按这个顺序执行：

1. 先判断任务属于**题库/分类优化**还是**分数异常诊断**  
2. 若是分类优化，默认只改 [assets/puzzles.json](../assets/puzzles.json)  
3. 进入**整类闭环**：先找出当前分类全部答案  
4. 进入**逐词闭环**：一次只处理一个答案  
5. 先查硬规则，再做修改  
6. **最小改动优先**，单槽能修就不整组重写  
7. 每改一题，必须先回读当前答案并立即重检  
8. 当前答案零错误后，记录 7 条 hints 的维度与可猜度判断  
9. 当前答案闭环后，才进入下一个答案  
10. 整个分类全部闭环后，再跑分类 / 全局验证  
11. 全部通过后，才统一向用户汇报结果  
12. 如果中途发现问题更像 override / calibration / controller，就切换到诊断流，而不是盲改 hints

再压缩成一句：

> **先路由，后闭环；先规则，后验证；逐词收口，整类收口；零错误前，不报完成。**
