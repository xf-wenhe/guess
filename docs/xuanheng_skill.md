## 0. 任务路由判断

当用户说 **"玄衡，帮我优化 xxx 分类"** 时，先判断任务类型。

### 0.1 默认判定

该语句默认属于 **题库/分类优化任务**，不是"分数异常诊断任务"。

也就是说，不会一上来就去查 calibration、后处理、override，除非用户同时表达了类似：

- "这个分类分数很离谱"
- "为什么无关词高分"
- "是不是 calibration 抬分"
- "这个百分比不合理"

如果只是"优化分类"，直接进入 `assets/puzzles.json` 的分类优化流。

---

## 1. 界定任务边界

不会直接动手改，而是先确认 3 件事：

| 确认项 | 说明 |
|--------|------|
| **目标分类** | 神话、美食、工作、动漫、生活…… |
| **任务范围** | 整个分类全部题目 / 分类里的一部分 / 只关注某类问题 |
| **改动边界** | 默认只改 `assets/puzzles.json`，不碰模型、校准、controller |

---

## 2. 进入"整类闭环"模式

只要说的是"优化 xxx 分类"，不会按"修一题就汇报"的方式做。

### 2.1 分类级闭环

- 找出该分类下全部目标题目
- 必须把这一批题目都处理完
- 全部通过后再统一告知结果

> 不会出现"改 1 题 → 说差不多好了"的情况，这在玄衡流程里是不合格的。

---

## 3. 对每个答案做"逐词闭环"

这是核心中的核心。按一个答案一个答案处理，不会整类粗改一遍再回头检查。

### 3.1 回读当前答案的原始数据

先读该答案在 `assets/puzzles.json` 里的内容：

- `answer`
- `category`
- 当前 7 条 hints

目的是先理解当前题到底长什么样，而不是直接凭印象改。

### 3.2 硬规则检查（第一轮，优先级最高）

| 检查项 | 说明 |
|--------|------|
| **字面隔离** | 每条 hint 是否包含 answer 里的任意字 |
| **同题重复** | 7 条 hints 之间是否重复，是否只是近义改写、语义骨架一样 |
| **维度分散** | 7 条是否尽量落在 7 个不同主维度，是否存在两个槽位其实在讲同一件事 |
| **模板词 / 元描述 / 残片** | 有没有"关键线索""情境线索""阶段提示"等元话语；有没有"物品""用品""所需"等空泛模板词；有没有不完整、读不通的残片 |
| **泄露检查** | "分类 + 前 6 条"是否已经足够锁词 |
| **锚点检查** | 第 7 条是不是最强锚点；第 6/7 条是不是泛词、弱相关词、假锚点 |

### 3.3 问题分型

先给答案归类问题，不会一边想一边乱改。不同问题改法完全不一样：

| 问题类型 | 特征 |
|----------|------|
| **前位泄露** | 第 2-6 条中的某条过强 |
| **高位锚点无效** | 第 6/7 条不够收束 |
| **维度重复** | 多条提示实际上在讲同一维度 |
| **自然度不足** | 生硬、模板化、像说明文而不是自然线索 |
| **字面污染** | 和答案共字 |
| **复用问题** | 同题 / 同类 / 全局重复 |

---

## 4. 修改原则：最小改动优先

不会默认整组重写。

### 4.1 优先改单槽

如果问题集中在一两条 hint：

- 只改当前有问题的槽位
- 尽量不动其他槽位
- 保持整体结构稳定

**示例：**

- 第 3 条太强 → 先找更弱的同功能替换
- 第 7 条太虚 → 先换成更强锚点
- 第 5 条和第 4 条维度撞车 → 优先改其中一个

### 4.2 什么时候才重写整组

只有在以下情况下才考虑整组重写：

- 7 条整体风格失控
- 多条维度重复
- 阶梯整体错乱
- 题目本身 hints 质量整体太差，局部修补收益很低

---

## 5. 改写设计原则

### 5.1 强度阶梯原则

7 条 hints 理解为一个从弱到强的梯度：

| 槽位 | 目标强度 |
|------|---------|
| slot 1 | ≈ 30% |
| slot 2 | ≈ 40% |
| slot 3 | ≈ 50% |
| slot 4 | ≈ 60% |
| slot 5 | ≈ 70% |
| slot 6 | ≈ 80% |
| slot 7 | ≈ 90% |

"百分比"不是字面相似度，而是 **"已知分类 + 该条 hint"后，玩家自然联想到答案的概率强度**。

> 玄衡规则里，自然表达优先于机械贴点。不会为了硬贴 30/40/50/60/70/80/90 去写很假很僵的词。

### 5.2 前 6 条不能锁词

前 6 条的职责不是"提前猜出来"，而是"逐步缩小空间"。

严格检查：

- 单独某条是不是过强
- 前几条组合是不是已经太明显
- "分类 + 前 6 条"是不是已经锁词

如果是，必须回退重写。

### 5.3 第 7 条必须承担收束职责

第 7 条是最后的最强锚点，应该：

- 最高辨识度
- 与答案机制直连
- 真正起到最终收束作用

> 特别防止：第 7 条只是个泛词、和很多答案都能搭、还不如第 5 条强——这都不合格。

### 5.4 维度必须分散

尽量让 7 条 hints 落到不同主维度。

**常见维度：**

时间 / 空间 / 场景 / 人物·对象 / 动作·机制 / 功能·用途 / 结果·反馈 / 情绪·氛围 / 观察视角 / 典型搭配·文化联想

**判断标准：**

不是换个近义词就算新维度。如果语义骨架一样，仍然算重复。比如：

- 都在讲"出现的地点" → 重复
- 都在讲"行为方式" → 重复
- 都在讲"视觉特征" → 重复

### 5.5 自然语义优先

写 hints 时优先保证：

- 母语者单独读这一条也能理解
- 是真实可感知、可联想的自然语义
- 不像模板说明书，不像强行压缩出来的残片

**口径控制：**

- 通常 5 字左右，必要时可放宽到 6 字
- 不会为了字数好看而牺牲自然性
- 不会强行全都做成 4 字格

### 5.6 不臆造

不会为了把梯度补齐，瞎写没有依据的词。如果缺少常识依据、缺少常见表达支持、只是"我觉得好像可以"——不会写进去。

> 证据不足，宁可中性降强度，也不编造。

---

## 6. 改后立即复检当前答案

这是玄衡流程最严格的一步。改了 1 条 hint 后，下一步不是继续想别的题，而是立刻复检当前答案。

### 6.1 复检清单

- [ ] 是否仍与 answer 零字符重叠
- [ ] 是否没有同题重复
- [ ] 是否没有模板词 / 元描述 / 残片
- [ ] 是否没有明显维度重复
- [ ] 是否没有把前 6 条变得更强
- [ ] 是否仍然保持第 7 条最强
- [ ] 是否整体更自然，不是更僵

> 如果当前答案还没复检过，不会认为它"已处理完"。

---

## 7. 当前答案通过后才进入下一个答案

一个答案完成闭环，至少表示：

1. 当前答案 7 条已经回读
2. 硬规则已检查
3. 修改已经完成
4. 修改后已经复检
5. 没有明显未解决的违规项

---

## 8. 整个分类做完后的验证

不是只看单题。

| 验证级别 | 脚本 |
|----------|------|
| 单题硬闸 | `python3 scripts/xuanheng_check_answer_strict.py <答案>` |
| 单题闭环 | `python3 scripts/xuanheng_check_answer_strict.py <答案> --review-json <review.json>` |
| 单题兼容补充 | `xuanheng_check_answer.py` |
| 分类级 | `xuanheng_check_detailed.py` |
| 全局规则级 | `scripts/validate_global_hint_rules_v1.py` |
| 强度/可猜度展示（按需） | `scripts/score_category_semantic_guess_v1.py` |

---

## 9. 中途切换到"诊断流"

虽然"优化分类"默认走题库流，但如果在处理中发现：

- 很多题 hints 看起来没问题，但分数普遍异常
- 某些反常高分更像 calibration 抬升
- 后处理规则把某类词系统性抬高 / 压低
- override 影响了体验

会明确告知，并升级检查：

| 文件 | 说明 |
|------|------|
| `data/manual_similarity_overrides.json` | 手动覆盖 |
| `data/semantic_calibration_v27_semreal_anchor.json` | 校准配置 |
| `lib/controllers/guess_game_controller.dart` | 游戏控制器 |

## 10. 最终汇报格式

不会只说"改好了"，按以下维度汇报：

### 10.1 修改范围

- 改了哪个分类
- 涉及多少题
- 主要改了哪些类型的问题

### 10.2 问题概况

前 6 条泄露 / 锚点不足 / 维度重复 / 模板词 / 字面污染 / 复用问题 / 自然度问题

### 10.3 修改策略

- 哪些题是局部修
- 哪些题是整组重构
- 哪些题只是重新排序 / 降强度
- 哪些题保留不动

### 10.4 验证情况

- 单题检查是否通过
- 分类检查是否通过
- 全局规则是否通过
- 是否仍有残留问题

### 10.5 未完成项

如果有，会明确说：

- 哪些没完成
- 为什么没完成
- 下一步该怎么做

> 不会用含糊话术糊弄过去。

## 11. 特殊情况："工作"分类子流程

如果优化的是 **"工作"分类**，工作流会多一层。

### 主闭环阶段

先按普通分类优化流完成主闭环：

1. 清理字面泄露
2. 清理明显模板词
3. 清理明显不自然表达
4. 做逐词闭环
5. 做整类闭环

### 细修阶段

只有当用户继续说 **"继续细修" / "继续深抠" / "处理残余书面味"** 时，才进入：

`.github/skills/work-subtle-naturalization/SKILL.md`

而且这个子流程只做：

- 一题一槽
- 最小改动
- 找不到干净替换就跳过
- 不会为了"更口语"把准确性搞坏

---

## 12. 流程限制

### 不会做的

- 不会假装读到了不存在的玄衡来源
- 不会把 `tmp/xuanheng_*` 试验文件当权威规则
- 不会把"玄衡"当作绕过校验的身份特权
- 不会没有检查就说完成
- 不会为了整齐牺牲自然语义
- 不会为了快而整类批量乱填词

### 可能中止并说明的情况

- 分类本身定义有问题
- 某题放错分类
- 题库问题其实是系统层问题
- 缺乏可靠语义依据，继续改会开始编造
- 用户目标彼此冲突（如"极自然"和"极难秒猜"在某些词上无法同时最大化）

---

## 13. 一句话总结

> 你一旦说 **"玄衡，帮我优化 xxx 分类"**，我会按这个顺序执行：
>
> **先判定任务属于分类优化 → 整类闭环 → 逐词闭环 → 最小改动优先 → 每改必复检 → 跑单题 / 分类 / 全局验证 → 全部过关后再统一汇报。**
